## 1. 터미널, 쉘, Bash 란?

### 컴퓨터에 명령을 내리는 두 가지 방법

| 방식 | 설명 |
|---|---|
| GUI | 마우스로 아이콘 클릭, 폴더 열기 등 (우리가 일상적으로 쓰는 방식) |
| CLI | 텍스트로 직접 명령어를 입력하는 방식 → 터미널과 쉘이 여기에 해당 |

### 터미널이란?

- **명령어를 입력하는 창** (화면/도구)
- 실제로 명령을 실행하지는 않고, 입력과 출력을 보여주는 역할
- Mac: `Terminal.app`, VS Code: 하단 터미널 창

### 쉘이란?

- **터미널 안에서 명령어를 해석하고 실행하는 프로그램**
- 터미널이 "전화기"라면, 쉘은 "전화기 너머의 통역사"

```
터미널 = 전화기 (소리를 전달하는 도구)
쉘     = 전화기 너머의 통역사 (말을 이해하고 실행하는 존재)
```

### 쉘의 종류

| 쉘 | 기본 사용 환경 |
|---|---|
| `bash` | Linux 기본값, 가장 널리 쓰임 |
| `zsh` | Mac 기본값 (최신 macOS) |
| `sh` | 가장 기본적인 쉘 |

### .bashrc / .zshrc 란?

- **쉘이 시작될 때 자동으로 실행하는 나만의 설정 파일**
- bash → `~/.bashrc`, zsh(Mac) → `~/.zshrc`

```
터미널 앱 열기
    └── zsh 시작
            └── ~/.zshrc 자동 실행
                    └── 거기 써둔 설정들이 자동 적용됨
```

## 2. export 명령어란?

### 쉘 변수 vs 환경 변수

```bash
MY_KEY=hello         # 쉘 변수: 현재 쉘에서만 유효
export MY_KEY=hello  # 환경 변수: 자식 프로세스에게도 전달됨
```

### 부모-자식 프로세스 개념

쉘에서 Python이나 Node 스크립트를 실행하면, 그 프로세스는 현재 쉘의 **자식 프로세스**로 실행된다.

```
현재 쉘 (부모)
│
├── 쉘 변수 (export 없음) → 이 공간에만 존재
│
└── 환경 변수 (export 있음) → 자식 프로세스에게 전달됨
        │
        ▼
    python script.py  ← 자식 프로세스
    node app.js       ← 자식 프로세스
```

**핵심:** `export` 없이 선언한 변수는 자식 프로세스(Python, Node 등)에서 읽을 수 없다.

```bash
# Python에서 os.getenv("MY_KEY") 를 실행하면
MY_KEY=hello         # → None (읽지 못함)
export MY_KEY=hello  # → "hello" (읽을 수 있음)
```

### 예외: dotenv 라이브러리

| 방식 | export 필요 여부 |
|---|---|
| `source ~/.env` 후 스크립트 실행 | ✅ 필요 |
| `python-dotenv`, `dotenv` 라이브러리 | ❌ 불필요 (파일을 직접 파싱하기 때문) |

## 3. source 명령어란?

> **"파일의 내용을 가져와서 현재 쉘에서 실행해줘"**

단순히 읽기만 하는 게 아니라 **현재 쉘에서 실행**한다는 것이 핵심.

### bash 실행 vs source 실행 비교

```bash
bash ~/wesley-apis/.env    # 자식 프로세스에서 실행 → 변수가 사라짐
source ~/wesley-apis/.env  # 현재 쉘에서 실행 → 변수가 남음
```

```
# bash로 실행할 때
현재 쉘 (부모)
│
└── bash ~/wesley-apis/.env  ← 자식 프로세스에서 실행
        export OPENAI_API_KEY=sk-xxxx  ← 자식 안에서만 적용되고 사라짐 ❌

# source로 실행할 때
현재 쉘 (부모)
│
└── source ~/wesley-apis/.env  ← 현재 쉘에서 바로 실행
        export OPENAI_API_KEY=sk-xxxx  ← 현재 쉘에 적용됨 ✅
```

| 방식 | 실행 위치 | 변수 남음? |
|---|---|---|
| `bash ~/file.env` | 자식 프로세스 | ❌ |
| `source ~/file.env` | 현재 쉘 | ✅ |

## 4. 홈 디렉토리 .env 파일로 API 키 관리하기

### 구조

```
~/ (홈 디렉토리)
├── wesley-apis/
│   └── .env          ← API 키 모아두는 마스터 파일
└── .zshrc            ← 터미널 시작 시 자동 실행되는 설정 파일
```

### Step 1. API 키 파일 만들기

```bash
# ~/wesley-apis/.env

# 깃허브 API KEY
export GITHUB_API_KEY=aaa

# 오픈라우터 API KEY
export OPEN_ROUTER_KEY=bbb

# 클로드 API KEY
export ANTHROPIC_API_KEY=ccc
```

### Step 2. .zshrc에 등록하기

```bash
echo 'source ~/wesley-apis/.env' >> ~/.zshrc
```

`>>` 는 파일 맨 끝에 내용을 추가하는 연산자.

### Step 3. 적용하기

```bash
source ~/.zshrc   # 터미널을 재시작하지 않고 바로 적용
```

### 전체 흐름

```
터미널 앱 열기
    └── zsh 시작
            └── ~/.zshrc 자동 실행
                    └── source ~/wesley-apis/.env 실행
                            └── API 키들이 환경변수로 등록됨 ✅
```

## 5. 쉘 파일 기본 문법 (.env / .zshrc용)

### 변수 선언

```bash
NAME=Wesley          # = 양쪽에 공백 없어야 함!
export API_KEY=abc123
```

### 변수 참조

```bash
echo $NAME           # Wesley
echo ${NAME}         # 동일, 중괄호는 명확하게 구분할 때 유용
echo "${NAME}_test"  # Wesley_test
```

### 주석

```bash
# 이건 주석이에요
```

### 다른 변수 참조해서 조합

```bash
BASE_URL=https://api.example.com
ENDPOINT=${BASE_URL}/v1/chat   # https://api.example.com/v1/chat
```

### 명령어 실행 결과를 변수에 저장

```bash
TODAY=$(date)        # 오늘 날짜를 변수에
HOME_DIR=$(pwd)      # 현재 경로를 변수에
```

### 자주 하는 실수

```bash
NAME = Wesley    # ❌ = 양쪽에 공백 있으면 에러
NAME=Wesley      # ✅

echo $NAME_test   # ❌ NAME_test 라는 변수로 인식됨
echo ${NAME}_test # ✅ NAME 변수 + "_test" 문자열

export GITHUB-API-KEY=aaa  # ❌ 하이픈(-) 사용 불가 (빼기 연산자로 해석됨)
export GITHUB_API_KEY=aaa  # ✅ 언더스코어(_) 사용
```

## 오늘의 핵심 요약

1. **터미널**은 명령어를 입력하는 창, **쉘**은 그 명령을 해석·실행하는 프로그램이다.
2. Mac의 기본 쉘은 `zsh`이며, 시작 설정 파일은 `~/.zshrc`이다.
3. `export` 없이 선언한 변수는 자식 프로세스(Python, Node 등)에서 읽을 수 없다.
4. `source`는 파일 내용을 **현재 쉘에서** 실행한다. `bash`로 실행하면 자식 프로세스에서 실행되어 변수가 사라진다.
5. `~/.zshrc`에 `source ~/wesley-apis/.env`를 추가하면, 터미널을 열 때마다 API 키가 자동으로 환경변수로 등록된다.
6. 환경변수 이름에는 **하이픈(-) 대신 언더스코어(_)** 를 써야 한다.
